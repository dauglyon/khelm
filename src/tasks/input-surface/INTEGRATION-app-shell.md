# Integration: app-shell -> input-surface

## Imports Required

### Layout Slot
- The input surface renders inside the **Toolbar** region (64px, below header, full width)
- `WorkspaceLayout` / `Toolbar` component provides the slot where the input bar is mounted
- The input bar replaces the toolbar's placeholder content ("Input surface placeholder")
- The toolbar is a fixed-position region -- the input surface must not alter the 64px height constraint

### Session Context
- **`sessionStore.activeSessionId`** -- the input surface includes `sessionId` in every card creation payload (see submit flow, `POST { type, content, mentions[], sessionId }`)
- Import the `useSession` hook or read `activeSessionId` from the Zustand session store to obtain the current session ID
- The input surface must not render or must disable submission when `activeSessionId` is null

### Environment Configuration
- **`getOllamaUrl()`** from `src/common/utils/env.ts` -- used by the classification service to determine the local Ollama endpoint for input classification
- **`getApiBaseUrl()`** from `src/common/utils/env.ts` -- used when the classifier falls back to API mode (Gemini/GPT-4.1 nano)

### API Layer (Orval + MSW)
- Card creation endpoint types and hooks generated by Orval from the OpenAPI spec
- The submit flow calls the card creation API; the generated mutation hook or fetch client provides type-safe request/response handling
- MSW handlers mock the card creation endpoint during development

### Auth Context
- **`authStore.token`** from `src/common/stores/authStore.ts` -- API requests for card creation and classification fallback require the auth token
- The input surface should be rendered only within authenticated routes (guaranteed by `RequireAuth` guard), but should handle token absence gracefully

### Provider Infrastructure
- **TanStack Query** `QueryClientProvider` -- the input surface uses TanStack Query for the card creation mutation (cache invalidation triggers workspace update)
- **LazyMotionProvider** -- the classification preview fade transitions (150ms, `easing.inOut`) rely on Motion being available via the provider tree

## Acceptance Criteria

1. Input bar renders inside the Toolbar layout region at `/session/:id` and does not exceed the 64px toolbar height
2. Submit flow reads `activeSessionId` from the session store and includes it in the card creation payload
3. Submit is disabled or prevented when no active session is set
4. Classification service reads `VITE_OLLAMA_URL` via `getOllamaUrl()` to connect to the local Ollama instance
5. Classification API fallback reads `VITE_API_BASE_URL` via `getApiBaseUrl()` for remote classifier endpoints
6. Card creation uses the Orval-generated types for the request payload -- no hand-written API types
7. Card creation mutation invalidates relevant TanStack Query caches (so workspace picks up the new card)
8. MSW mock for card creation endpoint returns realistic data during development
9. Auth token from `authStore` is included in API requests (card creation, classification fallback)
10. Input surface does not render on unauthenticated pages (relies on `RequireAuth` guard in the route tree)
