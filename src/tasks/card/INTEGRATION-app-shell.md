# Integration: app-shell -> card

## Imports Required

### Session Context
- **`sessionStore.activeSessionId`** -- every card record includes a `sessionId` field; the card domain reads the active session to scope card operations
- **`useSession(sessionId)`** hook -- card creation populates the `sessionId` field from the active session

### API Layer (Orval + MSW)
- Card CRUD endpoint types and hooks generated by Orval from the OpenAPI spec
- The card domain uses generated mutation hooks for card creation, update, and deletion
- The card domain uses generated query hooks for fetching individual card data and card chat history
- Orval-generated TypeScript types (`Card`, `CardType`, `CardStatus`, `CardContent`, `CardResult`) ensure type safety between frontend and backend
- MSW handlers mock all card endpoints during development, returning realistic Faker.js data matching the card data model

### Auth Context
- **`authStore.token`** from `src/common/stores/authStore.ts` -- card API requests (CRUD, chat) require authentication
- **`authStore.isAuthenticated`** -- the card domain should verify auth state before attempting API calls
- The card's `createdBy` field references the authenticated user ID

### Environment Configuration
- **`getApiBaseUrl()`** from `src/common/utils/env.ts` -- base URL for all card API endpoints (CRUD, chat, streaming)
- **`getWsUrl()`** from `src/common/utils/env.ts` -- WebSocket URL for receiving card status updates (`card:updated` events that drive status lifecycle transitions: thinking -> running -> complete/error)

### Provider Infrastructure
- **TanStack Query** `QueryClientProvider` -- the card domain uses TanStack Query for:
  - Fetching card data (detail view, chat history)
  - Card mutations (create, update, delete) with cache invalidation
  - Optimistic updates on card operations
- **LazyMotionProvider** -- card status animations (shimmer on thinking, spinner on running, checkmark scale-in on complete, shake on error) and `AnimatePresence` cross-fade transitions all require Motion in the provider tree

### Route Context
- **`useParams()`** from React Router -- the card detail view may be accessed via a route parameter; React Router provides param extraction
- The card domain renders within the `RequireAuth` + `RequireSession` guarded route tree

## Acceptance Criteria

1. Card creation includes the `sessionId` from `activeSessionId` in the session store
2. Card CRUD operations use Orval-generated types and hooks -- no hand-written API types
3. Card mutations invalidate relevant TanStack Query caches (session card list, individual card queries)
4. Card streaming content (SSE for running status) includes the auth token in request headers
5. Card chat transport (`fetch` + `ReadableStream`) uses `getApiBaseUrl()` for endpoint construction
6. Card status animations render correctly (Motion provider is available in the tree)
7. `AnimatePresence` cross-fade transitions between status states work as specified
8. MSW mocks provide realistic card data for all card types and statuses during development
9. Card chat MSW handlers support the streaming response pattern (SSE mock)
10. All card API requests include the auth token from `authStore`
